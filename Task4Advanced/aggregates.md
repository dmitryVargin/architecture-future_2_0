# Агрегаты доменной модели «Будущее 2.0»

## 1. Контекст «Управление пациентами»

### Агрегат: Пациент (Patient)

- **Корневая сущность:** `Patient`
- **Идентификатор:** `PatientId` (UUID)
- **Границы:** пациент, контактные данные, полис ОМС/ДМС, согласия на обработку данных
- **Инварианты:**
  - У пациента должен быть хотя бы один действующий полис
  - Согласие на обработку персональных данных обязательно (ФЗ-152)
  - ФИО и дата рождения не могут быть пустыми
  - СНИЛС уникален в системе
- **Жизненный цикл:** Зарегистрирован → Активен → Архивирован

### Агрегат: Медицинская карта (MedicalRecord)

- **Корневая сущность:** `MedicalRecord`
- **Идентификатор:** `MedicalRecordId` (UUID)
- **Границы:** записи в карте, диагнозы, назначения
- **Инварианты:**
  - Карта привязана ровно к одному пациенту (неизменяемая связь)
  - Записи в карте не удаляются — только добавляются (append-only)
  - Каждая запись содержит ID врача и дату
- **Жизненный цикл:** Создана → Активна (постоянное пополнение)

## 2. Контекст «Клинические операции»

### Агрегат: Расписание (Schedule)

- **Корневая сущность:** `Schedule`
- **Идентификатор:** `ScheduleId` (UUID, привязан к врачу + дате)
- **Границы:** слоты приёма, привязка к врачу, кабинет
- **Инварианты:**
  - Слоты не пересекаются по времени
  - Максимальная длительность слота — 120 минут
  - Врач не может иметь более одного расписания на один день
- **Жизненный цикл:** Черновик → Опубликовано → Закрыто

### Агрегат: Приём (Appointment)

- **Корневая сущность:** `Appointment`
- **Идентификатор:** `AppointmentId` (UUID)
- **Границы:** запись на приём, статус, результат приёма, назначения
- **Инварианты:**
  - Приём привязан к одному пациенту и одному врачу
  - Нельзя создать приём на занятый слот
  - Отменённый приём не может быть проведён
  - Результат приёма обязателен при статусе «Завершён»
- **Жизненный цикл:** Запланирован → В процессе → Завершён | Отменён | Неявка

## 3. Контекст «Медицинские исследования и ИИ»

### Агрегат: Исследование (Research)

- **Корневая сущность:** `Research`
- **Идентификатор:** `ResearchId` (UUID)
- **Границы:** назначение, тип исследования, биоматериалы, результаты
- **Инварианты:**
  - Исследование привязано к приёму или назначению врача
  - Результаты неизменяемы после подтверждения
  - Тип исследования должен существовать в справочнике
- **Жизненный цикл:** Назначено → В работе → Результат получен → Подтверждено

### Агрегат: ИИ-задача (AITask)

- **Корневая сущность:** `AITask`
- **Идентификатор:** `AITaskId` (UUID)
- **Границы:** входные данные (снимки, анализы), модель, результат inference, confidence score
- **Инварианты:**
  - Входные данные неизменяемы после создания задачи
  - Confidence score в диапазоне [0.0, 1.0]
  - Результат ИИ не является финальным диагнозом — требуется подтверждение врача
  - Версия модели фиксируется в момент создания задачи
- **Жизненный цикл:** Создана → В очереди → Выполняется → Завершена | Ошибка

## 4. Контекст «Финансовые операции»

### Агрегат: Счёт (Invoice)

- **Корневая сущность:** `Invoice`
- **Идентификатор:** `InvoiceId` (UUID)
- **Границы:** позиции счёта, скидки, привязка к пациенту/организации
- **Инварианты:**
  - Сумма счёта = сумма позиций − скидки
  - Счёт не может быть изменён после выставления (только сторнирование)
  - Минимум одна позиция в счёте
- **Жизненный цикл:** Черновик → Выставлен → Оплачен | Просрочен | Сторнирован

### Агрегат: Платёж (Payment)

- **Корневая сущность:** `Payment`
- **Идентификатор:** `PaymentId` (UUID)
- **Границы:** сумма, способ оплаты, привязка к счёту, транзакция
- **Инварианты:**
  - Сумма платежа > 0
  - Платёж привязан к существующему счёту
  - Сумма всех платежей по счёту не превышает сумму счёта
  - Способ оплаты из разрешённого списка
- **Жизненный цикл:** Инициирован → Обрабатывается → Подтверждён | Отклонён | Возврат

## 5. Контекст «Кредитование»

### Агрегат: Кредитный договор (CreditContract)

- **Корневая сущность:** `CreditContract`
- **Идентификатор:** `CreditContractId` (UUID)
- **Границы:** условия кредита, график платежей, начисления, статус
- **Инварианты:**
  - Процентная ставка фиксируется на момент подписания
  - График платежей генерируется автоматически при создании
  - Сумма кредита в пределах [min_amount, max_amount] продукта
  - Кредит не выдаётся без прохождения скоринга
- **Жизненный цикл:** Заявка → Скоринг → Одобрен → Выдан → Погашается → Закрыт | Просрочен | Реструктурирован

### Агрегат: Кредитная заявка (CreditApplication)

- **Корневая сущность:** `CreditApplication`
- **Идентификатор:** `CreditApplicationId` (UUID)
- **Границы:** данные заявителя, запрашиваемая сумма, скоринговый балл, решение
- **Инварианты:**
  - Заявитель должен быть идентифицирован (паспорт, ИНН)
  - Скоринговый балл рассчитывается только один раз на заявку
  - Решение (одобрено/отклонено) неизменяемо после принятия
- **Жизненный цикл:** Создана → На рассмотрении → Одобрена | Отклонена → Договор создан (если одобрена)

## 6. Контекст «Аналитика и BI»

### Агрегат: Датапродукт (DataProduct)

- **Корневая сущность:** `DataProduct`
- **Идентификатор:** `DataProductId` (UUID)
- **Границы:** метаданные, схема данных, SLA, владелец, контракт доступа
- **Инварианты:**
  - У каждого датапродукта есть владелец (Data Product Owner)
  - Схема данных версионируется — обратная совместимость обязательна
  - SLA на актуальность данных задан и контролируется
  - Доступ предоставляется только через контракт
- **Жизненный цикл:** Черновик → Опубликован → Обновляется → Deprecated → Удалён

### Агрегат: Отчёт (Report)

- **Корневая сущность:** `Report`
- **Идентификатор:** `ReportId` (UUID)
- **Границы:** конфигурация отчёта, источники данных, параметры фильтрации, расписание
- **Инварианты:**
  - Пользователь видит только данные в рамках своего уровня доступа
  - Источники данных — только опубликованные датапродукты
  - Расписание генерации не чаще 1 раза в минуту
- **Жизненный цикл:** Создан → Настроен → Активен → Архивирован

## 7. Контекст «Управление персоналом»

### Агрегат: Сотрудник (Employee)

- **Корневая сущность:** `Employee`
- **Идентификатор:** `EmployeeId` (UUID)
- **Границы:** личные данные, должность, подразделение, квалификация, сертификаты
- **Инварианты:**
  - Врач должен иметь действующий сертификат специалиста
  - Сотрудник привязан к одному подразделению
  - Табельный номер уникален
- **Жизненный цикл:** Принят → Активен → В отпуске → Уволен

## 8. Контекст «Инвентаризация»

### Агрегат: Единица инвентаря (InventoryItem)

- **Корневая сущность:** `InventoryItem`
- **Идентификатор:** `InventoryItemId` (UUID)
- **Границы:** наименование, категория, количество, местоположение, серийный номер
- **Инварианты:**
  - Количество на складе ≥ 0
  - Медицинское оборудование имеет дату следующей поверки
  - Лекарственные средства имеют срок годности
  - Серийный номер уникален для оборудования
- **Жизненный цикл:** Получено → На складе → Выдано → Списано | Возвращено
