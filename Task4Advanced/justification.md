# Обоснование событийного подхода для «Будущее 2.0»

## 1. Проблемы текущей архитектуры

### DWH как центральная точка интеграции

Сегодня DWH на базе SQL Server 2008 выполняет двойную роль: хранилище данных и интеграционная шина. Вся бизнес-логика сосредоточена в хранимых процедурах, а обмен данными между подразделениями (клиники, финтех, ИИ) идёт через DWH. Это создаёт:

- **Узкое горлышко**: добавление нового бизнес-направления (фармацевтика, производитель оборудования) требует изменений в центральном DWH
- **Медленная отчётность**: сложные отчёты формируются часы из-за сотен терабайт данных и множества трансформаций
- **Невозможность real-time**: пакетная обработка (batch ETL) не поддерживает near-real-time аналитику

### Apache Camel ESB как шина данных

- **Точка отказа**: все интеграции проходят через одну шину — сбой ESB останавливает все потоки
- **Синхронная связанность**: сервисы ожидают ответа от ESB, что создаёт каскадные задержки
- **Сложность маршрутизации**: со временем накопились сотни маршрутов, которые трудно тестировать и поддерживать
- **Масштабирование**: вертикальное, дорогое, с пределом

## 2. Преимущества событийного подхода

### Сравнительный анализ

| Критерий | Текущая архитектура (Camel + DWH) | Событийная архитектура (Kafka + Data Mesh) |
|----------|-----------------------------------|-------------------------------------------|
| **Связанность** | Высокая: все системы зависят от DWH и ESB | Низкая: домены взаимодействуют через события, не зная друг о друге |
| **Масштабируемость** | Вертикальная: наращивание мощности SQL Server | Горизонтальная: добавление партиций Kafka и инстансов сервисов |
| **Time-to-market** | Недели-месяцы: изменение DWH → тестирование → деплой всей системы | Дни: изменение одного домена, публикация нового события |
| **Отказоустойчивость** | Низкая: сбой DWH или Camel останавливает всё | Высокая: сбой одного домена не влияет на остальные; Kafka реплицирует данные |
| **Добавление нового домена** | Сложно: изменения в DWH-схеме, новые маршруты Camel, регрессионное тестирование | Просто: новый домен подписывается на нужные топики и публикует свои события |
| **Аналитика** | Batch: ETL → DWH → Power BI (часы задержки) | Near-real-time: Kafka → Flink/ClickHouse → Self-service BI (секунды-минуты) |
| **Регуляторное соответствие** | Все данные в одном месте — сложно разграничить доступ | Данные распределены по доменам — естественная изоляция по уровням доступа |
| **Стоимость изменений** | Высокая: каждое изменение затрагивает центральные компоненты | Низкая: изменения локализованы в пределах домена |

### Конкретные выгоды для «Будущее 2.0»

#### Интеграция фармацевтических компаний и производителя оборудования

При текущей архитектуре подключение нового партнёра требует:
1. Изменение схемы DWH (миграция, регрессионное тестирование)
2. Новые маршруты в Camel ESB
3. Доработка Power BI отчётов

При событийном подходе:
1. Новый домен публикует события в Kafka (`Pharma.DrugDelivered`, `Equipment.MaintenanceDone`)
2. Существующие домены подписываются на нужные события
3. Аналитика подхватывает новые данные автоматически через каталог датапродуктов

**Ожидаемое ускорение:** с 3-4 месяцев до 2-4 недель на интеграцию нового бизнес-направления.

#### Переход к near-real-time аналитике

Текущее состояние: формирование сложного отчёта занимает часы. Это критично для финтех-направления, где данные о транзакциях и кредитах должны быть актуальны.

Событийный подход: каждая финансовая операция генерирует событие → Kafka → stream processing (Flink) → материализованное представление в ClickHouse → дашборд обновляется за секунды.

**Ожидаемое улучшение:** время получения отчёта с часов до секунд для оперативной аналитики.

#### Независимость подразделений

Четыре подразделения «Будущего 2.0» (головной офис, клиники, ИИ-компания, финтех) развиваются с разной скоростью. Финтех выпускает новые фичи еженедельно, клиники — ежемесячно. Центральный DWH замедляет всех до скорости самого медленного.

Событийная архитектура позволяет каждому домену:
- Деплоиться независимо
- Выбирать собственный технологический стек
- Масштабироваться под свою нагрузку

#### Выход на новые регионы

При масштабировании на 2-3 новых региона событийная архитектура позволяет:
- Развернуть региональные кластеры Kafka с репликацией
- Соблюсти требования по локализации данных (медицинские данные в пределах региона)
- Подключить региональные клиники как новые источники событий без изменения центральных систем

## 3. Стратегия перехода

Полный отказ от Camel ESB и DWH невозможен одномоментно. Стратегия предусматривает поэтапный переход:

### Этап 1 (месяцы 1-6): Strangler Fig Pattern
- Новые интеграции идут через Kafka
- Camel ESB продолжает обслуживать существующие маршруты
- Anti-Corruption Layer транслирует события Camel → Kafka

### Этап 2 (месяцы 7-18): параллельная работа
- Критические потоки мигрируют на Kafka
- DWH продолжает работать для legacy-отчётности
- Новая аналитика строится на потоковых данных

### Этап 3 (месяцы 19-36): вывод legacy
- Camel ESB используется только как мост совместимости для оставшихся маршрутов
- DWH выводится из роли центрального хранилища
- Доменная аналитика полностью на событийной платформе

## 4. Риски событийного подхода и их митигация

| Риск | Митигация |
|------|-----------|
| Eventual consistency вместо strong consistency | Saga-паттерн для критических бизнес-процессов (финансовые транзакции); reconciliation-сервис |
| Сложность отладки распределённых потоков | Distributed tracing (OpenTelemetry); centralized logging; correlation ID в каждом событии |
| Дублирование обработки событий | Идемпотентность потребителей; exactly-once semantics в Kafka |
| Эволюция схем событий | Schema Registry с проверкой обратной совместимости; версионирование контрактов |
| Порядок обработки событий | Партиционирование по бизнес-ключу (patientId, contractId); ordering guarantees в пределах партиции |

## 5. Заключение

Событийный подход — не просто замена технологии интеграции. Это фундаментальное изменение способа взаимодействия доменов «Будущего 2.0», которое обеспечивает:

1. **Скорость** — time-to-market для новых бизнес-направлений сокращается в 3-5 раз
2. **Масштабируемость** — горизонтальное масштабирование вместо дорогого вертикального
3. **Устойчивость** — отказ одного домена не останавливает остальные
4. **Гибкость** — каждое подразделение развивается в своём темпе
5. **Аналитика** — переход от пакетной отчётности (часы) к near-real-time (секунды)

Эти преимущества критичны для компании, которая планирует масштабироваться по трём направлениям: продукты, география и данные.
