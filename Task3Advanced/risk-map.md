# Карта рисков трансформации «Будущее 2.0»

## Архитектурные риски

| # | Риск | Описание | Вероятность | Влияние | Триггеры | Меры снижения |
|---|------|----------|:-----------:|:-------:|----------|---------------|
| A1 | Потеря данных при миграции с DWH | Сотни терабайт данных в SQL Server 2008 содержат критичные медицинские и финансовые записи. Ошибки маппинга или неполная миграция приведут к потере целостности данных | Средняя | Высокое | Несовпадение схем, неполная документация legacy-структур | Параллельный запуск old/new с автоматическим сравнением контрольных сумм; поэтапная миграция по доменам; rollback-план на каждый этап |
| A2 | Чрезмерная связанность через Kafka | Замена Camel ESB на Kafka может воспроизвести те же проблемы, если не контролировать топологию топиков и контрактов | Средняя | Среднее | Более 50 топиков без Schema Registry; потребители без версионирования контрактов | Schema Registry с обязательной регистрацией; governance policy для топиков; ограничение fan-out через domain events |
| A3 | Vendor lock-in на Yandex Cloud | Привязка к managed-сервисам YC (Managed Kafka, Managed K8s, YDB) усложнит миграцию на другого провайдера | Низкая | Среднее | Санкционные или бизнес-риски; смена ценовой политики YC | Абстракция через Terraform и Kubernetes; минимальное использование проприетарных API; контейнеризация всех сервисов |
| A4 | Деградация производительности в переходный период | Антикоррупционные слои и двойная запись (DWH + новые сервисы) создадут дополнительную нагрузку | Высокая | Среднее | Запуск параллельной работы систем; рост объёма данных | Бюджетирование дополнительных ресурсов на переходный период; мониторинг latency на каждом ACL; приоритизация миграции самых нагруженных потоков |
| A5 | Несогласованность данных между доменами | В event-driven архитектуре eventual consistency может привести к временным расхождениям финансовых данных между доменами | Средняя | Высокое | Финансовая отчётность, требующая точной согласованности; регуляторные проверки | Saga-паттерн для критических бизнес-процессов; reconciliation-сервис для финансовых данных; SLA на время синхронизации между доменами |

## Организационные риски

| # | Риск | Описание | Вероятность | Влияние | Триггеры | Меры снижения |
|---|------|----------|:-----------:|:-------:|----------|---------------|
| O1 | Нехватка квалифицированных кадров | Переход на Kafka, Kubernetes, Go и event-driven подход требует компетенций, которых нет у текущей команды, привыкшей к SQL Server и PowerBuilder | Высокая | Высокое | Старт первого этапа; попытка параллельно поддерживать legacy и строить новое | Программа обучения за 3 месяца до старта; найм 2-3 Senior SRE/Platform Engineer; привлечение внешних консультантов на пилот |
| O2 | Сопротивление изменениям со стороны бизнеса | Операторы клиник и аналитики привыкли к Power BI и PowerBuilder; новый портал самообслуживания требует изменения привычек | Высокая | Среднее | Запуск портала самообслуживания; вывод Power BI | Champions-программа: обучение 2-3 «амбассадоров» в каждом подразделении; параллельная работа старых и новых инструментов; сбор обратной связи каждые 2 недели |
| O3 | Конфликт приоритетов между подразделениями | Четыре подразделения (головной офис, клиники, ИИ-компания, финтех) имеют разные приоритеты по данным и темпу трансформации | Средняя | Среднее | Распределение бюджета; выбор пилотного домена; конкуренция за ресурсы платформенной команды | Единый Architecture Board с представителями всех подразделений; прозрачная приоритизация через бизнес-ценность; квартальные OKR по трансформации |
| O4 | Потеря ключевых разработчиков legacy | Специалисты по PowerBuilder и SQL Server 2008 уникальны и могут уйти до завершения миграции | Средняя | Высокое | Объявление о выводе legacy из эксплуатации; рыночные предложения | Retention-программа на 18-24 месяца; документирование знаний legacy-систем; cross-training внутри команды |
| O5 | Несоблюдение регуляторных требований | Медицинские данные (ФЗ-152, приказы Минздрава) и финансовые данные (ЦБ РФ) имеют строгие требования к хранению и обработке | Средняя | Высокое | Миграция медицинских данных в облако; интеграция нового банковского ПО | Привлечение compliance-специалиста с первого этапа; аудит архитектуры на соответствие регуляторным требованиям; сертификация YC для работы с ПДн |

## Технологические риски

| # | Риск | Описание | Вероятность | Влияние | Триггеры | Меры снижения |
|---|------|----------|:-----------:|:-------:|----------|---------------|
| T1 | Отказ Kafka-кластера | Kafka становится единой точкой отказа для всей событийной платформы; потеря брокера приведёт к остановке межсервисного взаимодействия | Низкая | Высокое | Сбой диска; сетевая партиция; исчерпание ресурсов | Managed Kafka YC с 3+ брокерами в разных зонах доступности; настройка replication factor ≥ 3; алертинг на lag потребителей; fallback на sync API для критических операций |
| T2 | Несовместимость версий при миграции SQL Server 2008 | SQL Server 2008 использует устаревшие типы данных и хранимые процедуры, не имеющие аналогов в PostgreSQL | Высокая | Среднее | Начало миграции данных; обнаружение бизнес-логики в хранимых процедурах | Инвентаризация всех хранимых процедур и триггеров до миграции; автоматизированные тесты на эквивалентность результатов; pgloader с кастомными правилами маппинга |
| T3 | Масштабирование ML-inference под нагрузкой | ИИ-сервисы на Python могут не выдержать переход от пакетного к near-real-time режиму при росте числа клиник и регионов | Средняя | Среднее | Подключение 2-3 новых регионов; рост числа исследований в 3-5 раз | GPU-ноды в Managed K8s с автоскейлингом; батчинг inference-запросов; кэширование результатов для повторных диагнозов; рассмотрение ONNX Runtime для ускорения |
| T4 | Безопасность данных при переходе в облако | Перенос медицинских карт и финансовых данных в облако увеличивает поверхность атаки | Средняя | Высокое | Первая миграция данных в YC; интеграция с внешними системами | Шифрование at-rest (YC KMS) и in-transit (mTLS); RBAC на уровне Kubernetes namespace; VPC с приватными подсетями; SOC 2 аудит инфраструктуры; penetration testing |
| T5 | Непредсказуемые затраты на облачную инфраструктуру | Сотни терабайт данных и near-real-time обработка через Kafka могут привести к значительному превышению бюджета | Средняя | Среднее | Рост объёма данных; увеличение числа потребителей Kafka; хранение логов | FinOps-практики: теги на все ресурсы, еженедельный мониторинг затрат, committed use discounts; архивирование холодных данных в S3 Glacier; лимиты бюджета с алертами |
