# План управления рисками трансформации «Будущее 2.0»

## 1. Стратегия управления рисками

Управление рисками в проекте трансформации «Будущее 2.0» строится на трёх принципах:

1. **Проактивность** — выявление и снижение рисков до их наступления
2. **Непрерывность** — пересмотр карты рисков каждые 2 недели на Architecture Board
3. **Прозрачность** — все риски и статусы мер доступны всем стейкхолдерам

## 2. Матрица приоритизации

Риски приоритизируются по произведению вероятности и влияния:

| Приоритет | Уровень | Действие | Время реакции |
|:---------:|---------|----------|:-------------:|
| P1 | Высокая вероятность × Высокое влияние | Немедленные действия, эскалация на CTO | 24 часа |
| P2 | Средняя вероятность × Высокое влияние ИЛИ Высокая × Среднее | Активные меры снижения в текущем спринте | 1 неделя |
| P3 | Средняя × Среднее | Планируемые меры в ближайшем квартале | 1 месяц |
| P4 | Низкая × Среднее ИЛИ Низкая × Низкое | Мониторинг, действия при изменении статуса | Квартальный обзор |

## 3. Приоритизированный план действий

### P1 — Критические риски (немедленные действия)

#### O1: Нехватка квалифицированных кадров
- **Ответственный:** HR-директор + CTO
- **Срок:** месяцы 1-3 (до старта пилота)
- **Действия:**
  - Провести оценку текущих компетенций команды по матрице навыков
  - Запустить программу обучения: Kafka, Kubernetes, Go (внутренние и внешние тренинги)
  - Открыть 2-3 вакансии Senior Platform Engineer / SRE
  - Заключить контракт с внешним консультантом по event-driven архитектуре на 6 месяцев
- **Метрика контроля:** процент команды с сертификацией по целевым технологиям (цель: 60% к месяцу 6)

#### A1: Потеря данных при миграции с DWH
- **Ответственный:** Lead Data Engineer
- **Срок:** месяцы 1-6 (перед каждым этапом миграции)
- **Действия:**
  - Полная инвентаризация данных DWH: таблицы, связи, объёмы, критичность
  - Разработка автоматических тестов на целостность: контрольные суммы, row count, бизнес-правила
  - Параллельный запуск: новая система работает в shadow mode минимум 4 недели
  - Rollback-план с максимальным временем восстановления 4 часа
- **Метрика контроля:** 100% покрытие критичных таблиц тестами целостности; 0 инцидентов потери данных

### P2 — Высокие риски (активные меры)

#### O4: Потеря ключевых разработчиков legacy
- **Ответственный:** HR-директор + Тех. лид legacy-команды
- **Срок:** месяцы 1-24
- **Действия:**
  - Retention-бонусы для 5 ключевых специалистов по PowerBuilder/SQL Server (привязка к milestone-ам миграции)
  - Knowledge transfer sessions: 2 часа в неделю, запись на видео, документация в Confluence
  - Cross-training: каждый legacy-специалист обучает 2 новых инженеров своему домену
- **Метрика контроля:** документировано 100% критичных бизнес-процессов legacy; bus factor ≥ 2 для каждого компонента

#### O5: Несоблюдение регуляторных требований
- **Ответственный:** Compliance Officer + Security Lead
- **Срок:** месяцы 1-6 (до первой миграции данных)
- **Действия:**
  - Аудит архитектуры на соответствие ФЗ-152, требованиям ЦБ РФ и Минздрава
  - Выбор ЦОД YC с сертификацией для работы с ПДн и банковскими данными
  - Внедрение Data Classification: автоматическая маркировка данных по уровню конфиденциальности
  - Ежеквартальный аудит доступов и журналирования
- **Метрика контроля:** 0 нарушений при внешних проверках; 100% данных классифицировано

#### A5: Несогласованность данных между доменами
- **Ответственный:** Lead Architect
- **Срок:** месяцы 4-12
- **Действия:**
  - Реализация Saga-паттерна для кросс-доменных финансовых транзакций
  - Reconciliation-сервис: ежедневная автоматическая сверка балансов между доменами
  - SLA на eventual consistency: максимум 30 секунд для финансовых данных, 5 минут для аналитических
  - Dead Letter Queue с автоматическими алертами и ручной обработкой
- **Метрика контроля:** среднее время синхронизации < 10 секунд; количество расхождений при reconciliation < 0.01%

#### T4: Безопасность данных при переходе в облако
- **Ответственный:** CISO / Security Lead
- **Срок:** месяцы 1-36 (непрерывно)
- **Действия:**
  - Шифрование at-rest через YC KMS, in-transit через mTLS
  - Network: VPC с приватными подсетями, Security Groups, NAT Gateway
  - RBAC на уровне Kubernetes namespace + сервисных аккаунтов
  - Ежегодный penetration testing внешним подрядчиком
  - WAF и DDoS-защита для публичных эндпоинтов
- **Метрика контроля:** 0 критических уязвимостей по результатам pentest; MTTR для инцидентов безопасности < 2 часов

#### O2: Сопротивление изменениям
- **Ответственный:** Product Owner портала самообслуживания
- **Срок:** месяцы 6-18
- **Действия:**
  - Champions-программа: 2-3 «амбассадора» изменений в каждом подразделении
  - Параллельная работа Power BI и нового портала минимум 6 месяцев
  - Обучающие сессии для бизнес-пользователей (еженедельно в первые 3 месяца)
  - Канал обратной связи с SLA на ответ 24 часа
- **Метрика контроля:** NPS портала ≥ 7/10; доля пользователей, перешедших на новый портал > 80% к месяцу 18

### P3 — Средние риски (планируемые меры)

#### A2: Чрезмерная связанность через Kafka
- **Ответственный:** Lead Architect
- **Действия:** Schema Registry с обязательной регистрацией; governance policy; еженедельный review топологии топиков
- **Метрика:** количество топиков без Schema < 0; отсутствие циклических зависимостей между доменами

#### A4: Деградация производительности в переходный период
- **Ответственный:** SRE Lead
- **Действия:** бюджетирование +40% ресурсов на переходный период; мониторинг latency на ACL; автоскейлинг
- **Метрика:** p99 latency API < 500ms; p99 Kafka consumer lag < 1 минута

#### T2: Несовместимость версий SQL Server 2008
- **Ответственный:** Lead Data Engineer
- **Действия:** инвентаризация хранимых процедур; автоматические тесты эквивалентности; pgloader с кастомными правилами
- **Метрика:** 100% хранимых процедур промаппировано и протестировано до начала миграции домена

#### T3: Масштабирование ML-inference
- **Ответственный:** ML-инженер / ИИ-команда
- **Действия:** GPU-ноды с автоскейлингом; батчинг запросов; кэширование; нагрузочное тестирование
- **Метрика:** p95 inference latency < 2 секунд; автоскейлинг отрабатывает за < 3 минут

#### T5: Непредсказуемые облачные затраты
- **Ответственный:** FinOps Engineer / CFO
- **Действия:** теги на все ресурсы; еженедельный отчёт по затратам; committed use discounts; архивирование в S3 Glacier
- **Метрика:** отклонение от бюджета < 15%; стоимость хранения снижается на 30% через архивирование

### P4 — Низкие риски (мониторинг)

#### A3: Vendor lock-in на Yandex Cloud
- **Ответственный:** Lead Architect
- **Действия:** мониторинг зависимости от проприетарных API; ежеквартальный review
- **Метрика:** доля проприетарных API < 10% от общего числа интеграций

#### T1: Отказ Kafka-кластера
- **Ответственный:** SRE Lead
- **Действия:** replication factor ≥ 3; chaos engineering тесты; runbook для восстановления
- **Метрика:** RPO < 0; RTO < 15 минут; ежеквартальные disaster recovery учения

## 4. Процесс мониторинга

| Активность | Периодичность | Участники |
|------------|:-------------:|-----------|
| Обзор карты рисков | 2 недели | Architecture Board |
| Обновление метрик контроля | 1 неделя | Ответственные за риски |
| Полный пересмотр рисков | 1 квартал | CTO + все стейкхолдеры |
| Ретроспектива реализовавшихся рисков | По факту | Вся команда трансформации |

## 5. Эскалация

```
Уровень 1: Ответственный за риск → Lead Architect
Уровень 2: Lead Architect → CTO
Уровень 3: CTO → CEO (для рисков с влиянием на бизнес > 10 млн руб.)
```
